# 작업 규칙

- 변경 전에는 현재 저장소 구조와 기존 코드를 먼저 조사한다.
- `frontend/`와 `backend/`의 책임과 의존성 경계를 지킨다.
- 기존 Lovable UI를 임의로 삭제하거나 전면 교체하지 않는다.
- OpenAI API 키를 `frontend/`에 넣지 않는다.
- 비밀키와 `.env` 파일을 커밋하지 않는다.
- 기능 구현 후 관련 `test`, `lint`, `build`를 실행한다.

## 백엔드 작업 규칙

- 백엔드 API는 `frontend/`의 실제 라우트·타입·상태 요구를 근거로 설계하고, 변경 전 `docs/FRONTEND_BACKEND_MAPPING.md`를 갱신한다.
- OpenAI 등 서버 전용 API 키는 `backend/` 실행 환경 또는 배포 시크릿에서만 읽고, 응답·로그·문서에 원문 값을 노출하지 않는다.
- 외부에서 수집한 연구실·교수·논문·입시 정보에는 출처 URL과 마지막 확인 시점을 저장한다. 확인되지 않은 정보는 fixture 또는 mock으로 명시한다.
- LLM 생성 요약·초안·추천 근거는 원문에서 추출한 사실과 구분하고, 사용자가 실제 발송 또는 의사결정 전에 검토할 수 있게 한다.
- 해커톤 MVP는 단순한 동기 SQLAlchemy 접근과 SQLite 개발·테스트 / PostgreSQL 운영 호환 범위를 우선한다. 비동기 처리·크롤링 작업·외부 스토리지는 필요성이 확인된 뒤 도입한다.
